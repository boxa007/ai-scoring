# Агент-рекрутер Claude Code)

Той самий кейс, що n8n-агент, але **кодом**. Запускаєш → даєш вакансію → даєш список LinkedIn-посилань → код по кожному дістає профіль (через endpoint) → **гібридний скоринг** (код рахує арифметику, LLM судить за смислом) → ранжований список «кого запросити».

**Навіщо два треки:** n8n - візуально, для не-кодерів. Код - повний контроль, своя логіка, своя арифметика. На уроці показуємо обидва, щоб ти обрав свій.

---

## Потік (як це працює)

```
запускаєш score_candidates.py
  → даєш вакансію (файл або текст)
  → даєш список LinkedIn-URL кандидатів
  → КОД по кожному URL дьоргає endpoint apify-get-profile → JSON профілю
  → ГІБРИДНИЙ скоринг:
       • КОД: парс, роки досвіду, must-have відсів, зважена сума
       • LLM: бал по критерію + цитата-доказ (читає смисл)
  → ранжований shortlist «кого запросити» + чернетка повідомлення
```

## Швидкий старт

### Варіант А - з готовими профілями (без endpoint, найпростіше)

```bash
# профілі вже у sample-data/ у форматі apify-get-profile
python3 score_candidates.py \
  --job sample-data/job-backend.md \
  --profiles sample-data/ \
  --threshold 70 \
  --out shortlist.md
```

Маєш побачити: Олег (сильний) - ПРИГЛАСИТИ ~90, Ірина (frontend) - МОЖЛИВО/відсів.

### Варіант Б - з реальними LinkedIn-посиланнями (через endpoint)

Якщо у тебе є робочий webhook profigent (apify-get-profile):
```bash
# 1. дістати профілі по посиланнях у папку _raw/
source ../../scripts/lib-linkedin.sh   # шлях до lib-linkedin.sh у проєкті
mkdir -p _raw
li_fetch_profile "https://www.linkedin.com/in/SLUG1/" > _raw/c1.json
li_fetch_profile "https://www.linkedin.com/in/SLUG2/" > _raw/c2.json
# (пауза 1-2 сек між запитами - endpoint троттлить)

# 2. проскорити
python3 score_candidates.py --job моя-вакансія.md --profiles _raw/ --threshold 70
```

> Немає webhook? Працюй за Варіантом А: вставляй текст профілю руками у JSON-файл формату `sample-data/candidate-1.json`. ДЗ можна зробити і так.

## LLM-бекенд (автовизначення)

Скрипт сам обирає, чим думати:
1. **`claude` CLI** у PATH → використає його (нічого не треба налаштовувати)
2. інакше **`OPENAI_API_KEY`** у env → OpenAI gpt-4o-mini
   ```bash
   export OPENAI_API_KEY=sk-...
   ```
3. інакше **DRY-режим** → рахує тільки код-частину (заглушка по критеріях, позначка `[LLM OFF]`). Для тесту потоку без ключа.

Модель змінюється: `export SCORER_MODEL=gpt-4o` (default gpt-4o-mini).

---

## Чому це АГЕНТ-НА-КОДІ, а не скрипт і не чистий LLM

| | Чистий скрипт | Чистий LLM | **Наш гібрид** |
|---|---|---|---|
| Роки досвіду | рахує (точно) | бреше | **код рахує** ✓ |
| «розробляв на Пайтоні» = Python | НЕ бачить | бачить | **LLM бачить** ✓ |
| Must-have відсів | жорстко | «на думку» | **код жорстко** ✓ |
| Цитата-доказ | нема | вигадує | **LLM з профілю** ✓ |
| Зважена сума | точно | бреше | **код рахує** ✓ |

Гібрид бере надійність коду + розуміння смислу від LLM. Це і є «агент-на-коді»: **код володіє потоком і числами, LLM дає тільки те, чого код не вміє - читати живий текст.**

## ⚠️ Етика (hard, з уроку)

Скрипт у кінці завжди друкує:
> **AI оцінив, рішення за вами.** Не для авто-відмови без людини (EU AI Act, GDPR ст.22). «МОЖЛИВО» обовʼязково дивиться людина.

AI-скоринг кандидатів - юридично high-risk. Агент дає РЕКОМЕНДАЦІЮ, рішення приймає людина. Не використовуй для авто-відмови.
